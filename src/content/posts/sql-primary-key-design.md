[지난 글](/posts/sql-constraints-not-null-default-check/)에서 NOT NULL, DEFAULT, CHECK 제약을 다뤘다. 제약 조건 중에서도 가장 핵심적인 것이 **기본 키(Primary Key)**다. 기본 키 설계는 인덱스 성능, 외래 키 참조, 확장성에 직접 영향을 주며, 처음 설계가 틀리면 나중에 바꾸기 매우 어렵다.

## 기본 키의 역할

기본 키는 테이블에서 각 행을 유일하게 식별하는 컬럼(또는 컬럼 조합)이다. 선언하면 자동으로 `UNIQUE + NOT NULL` 제약이 적용되고, 대부분의 DBMS에서 기본 키 컬럼에 자동으로 인덱스가 생성된다.

```sql
CREATE TABLE 고객 (
    고객ID   INTEGER PRIMARY KEY,   -- 단일 열 PK
    이름     VARCHAR(100) NOT NULL,
    이메일   VARCHAR(255) NOT NULL UNIQUE
);

-- 복합 PK (여러 열 조합)
CREATE TABLE 권한 (
    사용자ID INTEGER NOT NULL,
    리소스   VARCHAR(100) NOT NULL,
    PRIMARY KEY (사용자ID, 리소스)
);
```

기본 키는 외래 키 참조의 대상이 된다. 다른 테이블이 이 기본 키를 참조하므로, 기본 키 값은 변경되지 않는 것이 중요하다.

## 자연 키 vs 대리 키

![자연 키 vs 대리 키](/assets/posts/sql-primary-key-design-types.svg)

### 자연 키(Natural Key)

비즈니스 도메인에서 이미 유일성이 보장된 값이다.

```sql
-- 자연 키 예시
CREATE TABLE 국가 (
    국가코드  CHAR(2)      PRIMARY KEY,  -- 'KR', 'US', 'JP'
    국가명   VARCHAR(100) NOT NULL
);
-- 국가코드는 변경 없고 짧고 의미 있음 → 자연 키 적합

CREATE TABLE 고객 (
    이메일   VARCHAR(255) PRIMARY KEY,   -- 자연 키 (부적절!)
    이름     VARCHAR(100) NOT NULL
);
-- 이메일은 변경 가능 → 부적절
```

코드 테이블처럼 값이 짧고 변경되지 않으며 의미가 명확한 경우에는 자연 키가 적합하다. 주민등록번호처럼 개인정보인 경우에는 절대 기본 키로 사용하지 않는다.

### 대리 키(Surrogate Key)

시스템이 자동으로 생성하는 의미 없는 식별자다. 대부분의 테이블에서 권장하는 방식이다.

```sql
CREATE TABLE 고객 (
    고객ID   INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    이메일   VARCHAR(255) NOT NULL UNIQUE,  -- 비즈니스 식별자는 UNIQUE
    이름     VARCHAR(100) NOT NULL
);
```

이메일을 기본 키로 쓰지 않고 대리 키를 만들어 이메일은 `UNIQUE` 제약으로 보호한다. 이메일이 변경되어도 외래 키를 사용하는 다른 테이블에 영향이 없다.

## 자동 증가 키

![자동 증가 키 — SERIAL · IDENTITY · UUID](/assets/posts/sql-primary-key-design-autogen.svg)

### IDENTITY (SQL:2003 표준)

```sql
-- PostgreSQL (IDENTITY, 표준)
CREATE TABLE 주문 (
    주문ID   INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- PostgreSQL (SERIAL, 레거시)
CREATE TABLE 주문_old (
    주문ID   SERIAL PRIMARY KEY
);

-- MySQL
CREATE TABLE 주문 (
    주문ID   INT AUTO_INCREMENT PRIMARY KEY
);

-- Oracle (12c+)
CREATE TABLE 주문 (
    주문ID   NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);
```

`GENERATED ALWAYS`는 명시적으로 값을 삽입하려 하면 오류가 난다. `GENERATED BY DEFAULT`는 명시적 값도 허용한다. 마이그레이션 데이터를 넣을 때는 `BY DEFAULT`가 편리하다.

### UUID

```sql
-- PostgreSQL: UUID v4 (랜덤)
CREATE TABLE 이벤트 (
    이벤트ID UUID DEFAULT gen_random_uuid() PRIMARY KEY
);

-- UUID 값 예시: 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'
```

UUID v4는 완전 랜덤이라 순서가 없다. B-tree 인덱스에 삽입할 때 임의 위치에 삽입되어 **페이지 단편화**가 발생한다. 대규모 INSERT 성능이 순차 정수보다 30~50% 느릴 수 있다.

**UUID v7**(2023년 RFC 9562)은 앞 48비트에 밀리초 타임스탬프를 포함해 삽입 순서를 보장한다. PostgreSQL 17+는 `gen_random_uuid_v7()`을 기본 제공하고, 이전 버전은 `pgcrypto` 확장이나 직접 생성 함수로 구현한다.

## 상황별 기본 키 선택

| 상황 | 권장 PK |
|---|---|
| 단일 DB, 일반 서비스 | INTEGER IDENTITY (BIGINT: 20억 초과 예상) |
| URL에 ID 노출 | 내부는 INTEGER, 외부용 UUID를 별도 컬럼으로 |
| 분산 시스템, 마이크로서비스 | UUID v7 또는 Snowflake ID |
| 코드 테이블 (국가, 화폐) | 자연 키 (CHAR(2~3)) |
| 다대다 연결 테이블 | 복합 PK 또는 INTEGER IDENTITY + 복합 UNIQUE |

기본 키는 한번 정하면 변경 비용이 매우 크다. 설계 초기에 확장성과 보안(ID 노출 여부)을 함께 고려한다.
