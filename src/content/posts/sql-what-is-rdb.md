데이터를 컴퓨터에 저장하는 방법은 여러 가지가 있지만, 지난 50여 년간 소프트웨어 산업 표준 자리를 지켜온 방식이 바로 **관계형 데이터베이스(Relational Database, RDB)** 다. 오늘날 서비스를 운영하는 회사 중 RDB 없이 운영되는 곳은 거의 없다. SQL을 제대로 이해하려면 먼저 "왜 RDB인가"라는 질문에 답해야 한다.

## 파일 시스템의 한계

1960~70년대까지 기업들은 데이터를 단순 파일에 보관했다. 고객 데이터는 `고객.txt`, 주문 데이터는 `주문.txt`에 각각 저장한다. 처음에는 단순해 보이지만 규모가 커지면 세 가지 문제가 드러난다.

- **중복 저장**: 고객 이름이 여러 파일에 중복 저장되어 변경 시 모든 파일을 동기화해야 한다.
- **데이터 불일치**: 한 파일만 수정되면 같은 사람의 정보가 파일마다 달라진다.
- **공유 어려움**: 여러 프로그램이 같은 파일을 동시에 수정할 때 충돌이 발생한다.

## 관계형 모델의 등장

1970년 IBM의 에드거 F. 코드(Edgar F. Codd)가 논문 「A Relational Model of Data for Large Shared Data Banks」를 발표하며 관계형 모델이 탄생했다. 핵심 아이디어는 단순하다. **데이터를 2차원 표(테이블)로 표현하고, 표와 표 사이를 공통 속성(키)으로 연결한다.**

| 개념 | 수학적 용어 | 실제 의미 |
|---|---|---|
| 테이블(Table) | 릴레이션(Relation) | 행과 열로 이루어진 데이터 집합 |
| 행(Row) | 튜플(Tuple) | 하나의 레코드, 개체 인스턴스 |
| 열(Column) | 속성(Attribute) | 데이터의 한 특성, 데이터 타입 포함 |
| 기본 키(PK) | 식별자 | 각 행을 유일하게 식별하는 열 조합 |
| 외래 키(FK) | 참조 | 다른 테이블의 기본 키를 참조하는 열 |

## SQL로 데이터 조회하기

RDB의 진짜 힘은 **선언형 언어 SQL**에 있다. "어떻게 찾을지"가 아니라 "무엇을 원하는지"만 기술하면 DBMS가 최적의 실행 경로를 찾아준다.

```sql
SELECT c.이름, o.제품명, o.금액
FROM   고객 c
JOIN   주문 o ON c.고객ID = o.고객ID
WHERE  o.금액 >= 100000
ORDER  BY o.금액 DESC;
```

고객 테이블과 주문 테이블을 `고객ID` 키로 연결해 100,000원 이상 주문을 내림차순으로 가져온다. 배열을 순회하거나 파일을 직접 읽는 코드 없이 단 네 줄이면 된다.

## RDB를 선택하는 이유

**데이터 무결성**: `NOT NULL`, `UNIQUE`, `CHECK`, 외래 키 같은 제약 조건이 잘못된 데이터가 저장되는 것을 데이터베이스 계층에서 차단한다. 애플리케이션 코드에 의존하지 않아도 된다.

**ACID 트랜잭션**: 원자성(Atomicity), 일관성(Consistency), 격리성(Isolation), 지속성(Durability)을 보장한다. 계좌 이체 도중 장애가 나도 돈이 사라지거나 두 배가 되지 않는다.

**표준 SQL**: ISO/IEC SQL 표준 덕분에 MySQL에서 배운 쿼리가 PostgreSQL, Oracle에서도 대부분 동작한다. 특정 벤더에 종속되지 않는다.

**성숙한 생태계**: 수십 년간 쌓인 최적화 기법, 모니터링 도구, 마이그레이션 도구, 그리고 무엇보다 방대한 레퍼런스와 인재 풀이 있다.

## NoSQL과의 관계

NoSQL이 등장한 2010년대 이후 "RDB는 구식"이라는 오해가 퍼졌지만, 실제로는 목적이 다르다. 문서(Document) DB는 스키마 없는 자유로운 구조를 위해, 키-값(Key-Value) DB는 극단적인 읽기 성능을 위해 선택한다. 트랜잭션, 복잡한 관계, 분석 쿼리가 필요한 곳이라면 RDB가 여전히 최선이다. 대형 서비스조차 코어 데이터는 PostgreSQL이나 MySQL 위에서 관리하고 NoSQL을 캐시·검색 보조 용도로 병행한다.
