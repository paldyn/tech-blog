## 왜 파일에 저장하면 안 될까?

처음 프로그래밍을 배울 때 데이터를 CSV나 텍스트 파일에 저장한 경험이 있을 것이다. 간단한 프로젝트에서는 잘 동작한다. 그런데 사용자가 수만 명이 되고, 팀원이 동시에 데이터를 수정하고, "특정 조건에 맞는 데이터만 빠르게 찾아야 한다"는 요구사항이 생기면 파일 시스템은 빠르게 한계를 드러낸다.

데이터베이스는 그 한계를 해결하기 위해 만들어진 소프트웨어다. 단순히 "데이터를 저장하는 곳"이 아니라, **데이터를 안전하게·효율적으로·일관성 있게 관리하기 위한 시스템**이다.

---

## 파일 시스템의 다섯 가지 한계

![파일 시스템 vs 관계형 데이터베이스](/assets/posts/sql-what-is-rdb-filesystem-vs-db.svg)

### 1. 데이터 중복 저장

주문 파일, 고객 파일, 배송 파일—각각의 파일에 같은 고객 정보(이름, 주소, 전화번호)가 반복 저장된다. 고객이 전화번호를 바꾸면 세 파일을 모두 수정해야 한다. 하나라도 빠뜨리면 **불일치(inconsistency)**가 발생한다.

### 2. 쿼리 언어 없음

"주문 금액이 10만 원 이상이고, 2024년 1월에 가입한 고객의 주문 목록"을 가져오려면? 파일을 처음부터 끝까지 읽으면서 코드로 직접 필터링해야 한다. 데이터가 많아질수록 성능도, 코드 복잡도도 함께 올라간다.

### 3. 동시 접근 충돌

두 프로세스가 같은 파일을 동시에 수정하면 어떻게 될까? 운영체제 수준의 파일 락(lock)이 없다면 데이터가 뒤섞인다. 실제 서비스에서 이 문제는 재현이 어렵고, 디버깅은 더 어렵다.

### 4. 형식과 무결성 미보장

"생년월일" 컬럼에 "19900101", "1990-01-01", "90/1/1"이 혼재해도 파일은 아무 말을 하지 않는다. 빈 값을 허용할지, 어떤 범위의 숫자만 들어올 수 있는지—이 모든 것을 애플리케이션 코드가 직접 검증해야 한다.

### 5. 애플리케이션 종속

파일 구조를 알아야만 데이터를 읽을 수 있다. 파일 포맷을 바꾸면 그 파일을 읽는 모든 프로그램을 함께 수정해야 한다. **데이터와 로직이 강하게 결합**되어 있는 것이다.

## 관계형 데이터베이스(RDB)의 등장

1970년 IBM의 에드거 F. 코드(Edgar F. Codd)는 논문 *"A Relational Model of Data for Large Shared Data Banks"*에서 수학의 집합 이론을 기반으로 한 **관계형 모델(Relational Model)**을 제안했다. 이 모델의 핵심 아이디어는 단순하다:

> **데이터를 테이블(릴레이션)로 표현하고, 집합 연산으로 원하는 데이터를 추출한다.**

관계형 DBMS(RDBMS)는 이 모델을 구현하면서 파일 시스템의 다섯 가지 문제를 체계적으로 해결했다.

| 문제 | RDB의 해결책 |
|------|------------|
| 중복 저장 | 정규화(Normalization) + 외래 키(FK) 참조 |
| 검색 불편 | SQL 쿼리 언어 |
| 동시 접근 | 트랜잭션 + 잠금(Lock) 메커니즘 |
| 형식·무결성 | 스키마(Schema) + 제약 조건(Constraint) |
| 앱 종속 | 데이터 독립성(물리적/논리적) |

## 릴레이션, 튜플, 속성

RDB는 수학적 개념에서 이름을 빌려왔다. 처음엔 낯설지만 금방 익숙해진다.

![릴레이션 해부](/assets/posts/sql-what-is-rdb-table-structure.svg)

- **릴레이션(Relation)**: 우리가 흔히 말하는 **테이블(Table)**. 같은 구조를 가진 행들의 집합이다.
- **튜플(Tuple)**: **행(Row)**, 또는 레코드(Record). 한 명의 직원 정보 전체처럼 하나의 데이터 인스턴스다.
- **속성(Attribute)**: **열(Column)**. 데이터의 한 가지 특성(이름, 이메일 등)이다.
- **차수(Degree)**: 릴레이션의 속성 개수. 위 예에서 4.
- **카디널리티(Cardinality)**: 릴레이션의 튜플 개수. 위 예에서 3.

실무에서는 "테이블", "행", "열"이라는 표현을 더 많이 쓰지만, 공식 문서나 이론서에서는 관계형 모델 용어가 그대로 등장한다. 두 가지 표현을 모두 알아두는 것이 좋다.

## 데이터베이스 vs DBMS vs RDBMS

용어 혼동이 잦으니 한 번 정리하자.

```text
데이터베이스 (Database, DB)
  └─ 데이터의 집합 그 자체

DBMS (Database Management System)
  └─ DB를 관리하는 소프트웨어 (Oracle, PostgreSQL, MySQL 등)

RDBMS (Relational DBMS)
  └─ 관계형 모델을 구현한 DBMS
  └─ 현업에서 "DB"라고 하면 대부분 RDBMS를 의미

SQL (Structured Query Language)
  └─ RDBMS와 소통하기 위한 표준 언어
```

DBMS는 데이터를 저장하고, 검색하고, 수정하고, 삭제하는 기능뿐 아니라 **접근 제어, 백업, 동시성 제어**까지 담당한다. 우리가 앞으로 302편에 걸쳐 공부할 것들이 모두 DBMS가 하는 일이다.

## 간단한 SQL 맛보기

이론은 충분히 봤으니, 아주 간단한 SQL 두 줄로 마무리하자.

```sql
-- 테이블 만들기
CREATE TABLE employees (
  id      INTEGER PRIMARY KEY,
  name    VARCHAR(100) NOT NULL,
  email   VARCHAR(200) UNIQUE,
  dept_id INTEGER
);

-- 데이터 조회: 부서 10번 소속 직원
SELECT id, name, email
FROM   employees
WHERE  dept_id = 10;
```

`CREATE TABLE`은 릴레이션(테이블)을 정의한다. `SELECT … FROM … WHERE`는 원하는 튜플(행)을 조건으로 걸러낸다. 파일을 전부 읽고 코드로 필터링하던 것을 단 세 줄이 대체한다.

물론 이 두 구문 뒤에는 인덱스, 옵티마이저, 트랜잭션 등 엄청난 구조가 숨어 있다. 그것들을 한 편씩 파헤치는 게 이 시리즈의 목표다.

## 정리

- 파일 시스템은 단순 저장에는 충분하지만, 중복·무결성·동시성·검색의 문제를 스스로 해결하지 못한다.
- 관계형 데이터베이스는 **집합 이론 기반의 릴레이션 모델**로 이 문제들을 체계적으로 해결한다.
- 테이블 = 릴레이션, 행 = 튜플, 열 = 속성. 두 표현 모두 알아두자.
- DBMS는 저장 이상의 일(동시성 제어, 무결성 보장, 접근 제어)을 한다.
