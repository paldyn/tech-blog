[지난 글](/posts/nexacro-n-vs-platform/)에서 Nexacro Platform과 N이 코드 수준에서 무엇이 달라지는지 항목별로 비교했습니다. 이번에는 그 비교를 바탕으로 "그래서 우리 프로젝트에 Nexacro N을 써도 되는가?"라는 질문에 실질적으로 답해봅니다. 프레임워크 선택은 기술 스펙만으로 결정되지 않습니다. 팀 역량·업무 도메인·기존 자산·예산이 모두 얽혀 있으며, Nexacro N이 빛을 발하는 맥락이 분명히 존재합니다.

## Nexacro N이 강한 영역

Nexacro N의 핵심 강점은 **대규모 그리드 중심 업무 화면을 적은 코드로 일관성 있게 만드는 능력**입니다. 100개가 넘는 화면을 관리할 때 React나 Vue는 컴포넌트 간 일관성 유지를 개발자의 역량에 맡기는 반면, Nexacro N은 폼·컴포넌트·데이터 바인딩이 규격화된 구조로 제공됩니다. 화면이 많아질수록 이 규격화가 유지보수 비용을 낮춥니다.

또한 Dataset–Transaction 구조는 서버 통신 방식을 단순화합니다. HTTP 요청마다 직접 fetch를 쓰는 대신 `transaction()` 호출 하나로 여러 Dataset의 변경 사항을 서버로 일괄 전송하고, 응답 Dataset을 컴포넌트에 자동 반영합니다. CRUD 중심의 업무 시스템에서 이 패턴은 코드 라인을 극적으로 줄여줍니다.

## 도입 판단 매트릭스

![Nexacro N 도입 판단 기준](/assets/posts/nexacro-n-when-to-use-decision.svg)

판단 매트릭스에서 핵심 분기점 두 가지를 먼저 확인해야 합니다.

**첫째, IE 지원이 필수인가.** Nexacro N은 IE를 공식 지원하지 않습니다. 관공서나 금융 현장 중 일부는 여전히 IE11 환경의 특정 내부망 시스템을 운용합니다. 그런 환경이라면 N 전환 시점을 미루거나 Platform을 병행 유지해야 합니다.

**둘째, 팀에 Nexacro 경험이 있는가.** Nexacro는 학습 곡선이 있습니다. JavaScript에 능숙하더라도 Dataset·Transaction 패턴, Form 생명주기, 이벤트 체계를 처음부터 배우는 데 최소 2~4주가 필요합니다. 신규 팀 전원이 React/Vue 전문가라면 Nexacro N을 채택하는 비용이 생각보다 큽니다. 반대로 기존 Nexacro Platform 개발자가 팀에 있다면, N 전환은 비교적 완만합니다.

## 실전 도입 시나리오

![Nexacro N 핵심 도입 시나리오](/assets/posts/nexacro-n-when-to-use-scenarios.svg)

### 금융·보험 ERP

국내 금융권은 Nexacro의 가장 전통적인 고객군입니다. 계정계·여수신·보험 언더라이팅 등 수백 개 화면을 단일 배포 체계로 운영해야 하고, 실시간 계좌 잔액·거래내역을 수십만 행 그리드로 보여줘야 하는 요구가 많습니다. Nexacro N은 이 조건에서 Platform보다 더 빠른 렌더링과 더 넓은 브라우저 지원을 제공합니다.

### 공공·행정 시스템

전자정부 표준 프레임워크를 쓰는 공공 프로젝트에서 프런트엔드로 Nexacro N을 선택하는 사례가 늘고 있습니다. 조달청 등록 제품 요건, GS 인증, 벤더 유지보수 지원이 공공 발주 기준에 맞기 때문입니다. 결재·통계·공문 관리 같은 화면은 그리드 비중이 높아 N의 강점이 그대로 발휘됩니다.

### 제조 MES / SCM

공장 폐쇄망 환경에서 설비 모니터링·생산 실적·자재 BOM을 다루는 MES 시스템은 인트라넷 단독 배포가 기본입니다. Nexacro N의 HTML5 런타임은 별도 플러그인 없이 내부 웹서버에서 서비스됩니다. 폐쇄망 기업 환경에서 ActiveX 의존성이 사라진 것은 큰 장점입니다.

### Platform → N 점진적 이관

이미 Nexacro Platform으로 구축된 시스템을 N으로 이관하는 경우가 현재 가장 많은 비중을 차지합니다. `.xfdl` 형식의 기존 화면 파일을 Nexacro Studio N에서 직접 열 수 있고, 주요 API는 호환됩니다. IE 의존 코드(ActiveX 호출, IE 전용 DOM API)만 선별적으로 수정하면 대부분의 화면을 재사용할 수 있습니다. 전면 재개발보다 비용이 훨씬 낮습니다.

## Nexacro N이 맞지 않는 상황

도입을 재고해야 하는 신호도 명확합니다.

- **B2C 서비스**: 소비자 대상 쇼핑몰이나 콘텐츠 플랫폼은 SEO, 접근성, 모바일 UX가 중요합니다. Nexacro N은 이 영역에 최적화되어 있지 않습니다.
- **모바일 앱 전용**: Nexacro N은 모바일 브라우저에서 동작하지만 네이티브 앱 수준의 모바일 UX를 기대하면 실망스럽습니다. 모바일 중심 서비스라면 React Native나 Flutter가 낫습니다.
- **소규모 스타트업**: 라이선스 비용과 학습 곡선을 감당할 여력이 없는 초기 스타트업에는 오픈소스 기반 스택이 현실적입니다.
- **그리드가 없는 화면 위주**: 콘텐츠·미디어·마케팅 페이지처럼 그리드보다 디자인 자유도가 중요한 화면은 Next.js/Astro 같은 도구가 더 자연스럽습니다.

## 결정 흐름 정리

판단을 단순화하면 이렇습니다.

```text
업무 화면 100개↑ AND 그리드 복잡  →  Nexacro N 강력 추천
기존 Platform 자산 있음            →  N 이관 검토
IE 필수 지원 AND 단기 마감         →  Platform 유지 또는 병행
B2C / 모바일 / SEO 중요            →  Web SPA 스택 권장
```

이 기준은 절대적이지 않습니다. 팀 역량과 기존 계약·운영 체계에 따라 달라질 수 있으니, 파일럿 프로젝트로 2~3개 화면을 직접 구현해 보는 것이 가장 정확한 판단 방법입니다.
