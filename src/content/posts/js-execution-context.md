클로저, 스코프, 호이스팅, `this` 바인딩 — 이 개념들을 앞의 글들에서 하나씩 살펴봤습니다. 그런데 이 모든 현상에는 공통적인 기반이 있습니다. JavaScript 엔진이 코드를 실행할 때 내부적으로 만드는 **실행 컨텍스트(Execution Context)** 입니다.

실행 컨텍스트를 이해하면 "왜 호이스팅이 발생하는가", "왜 `let`은 `var`와 다르게 동작하는가", "클로저는 어떻게 바깥 변수를 기억하는가" 같은 질문에 각각의 단편적인 규칙이 아니라 하나의 통일된 원리로 답할 수 있게 됩니다.

---

## 실행 컨텍스트란

JavaScript 엔진은 코드를 실행하기 전에 실행을 위한 환경을 먼저 준비합니다. 이 환경 묶음이 실행 컨텍스트입니다. 세 가지 종류가 있습니다.

- **전역 실행 컨텍스트**: 스크립트가 처음 로드될 때 단 하나 생성됩니다.
- **함수 실행 컨텍스트**: 함수가 호출될 때마다 새로 생성됩니다.
- **Eval 실행 컨텍스트**: `eval()` 함수 내부 코드용. 사용을 피하는 것이 권장됩니다.

---

## 콜 스택 — 컨텍스트의 쌓임과 제거

함수를 호출할 때마다 새 실행 컨텍스트가 **콜 스택(Call Stack)** 에 push됩니다. 함수가 값을 반환하면 해당 컨텍스트는 pop됩니다. 후입선출(LIFO) 구조입니다.

```js
function bar() {
  return 42;
}

function foo() {
  return bar();
}

foo();
```

이 코드가 실행될 때 스택의 변화를 시각화하면 다음과 같습니다.

![실행 컨텍스트 스택의 push/pop 과정](/assets/posts/js-execution-context-diagram.svg)

전역 컨텍스트는 스크립트가 종료될 때까지 스택 맨 아래에 남아있습니다. `foo()`가 호출되면 그 위에 `foo` 컨텍스트가 올라가고, `foo` 안에서 `bar()`가 호출되면 다시 그 위에 `bar` 컨텍스트가 쌓입니다. `bar`가 반환하면 pop, 이어 `foo`도 반환하면 pop, 최종적으로 전역 컨텍스트만 남습니다.

## 실행 컨텍스트의 두 단계

각 실행 컨텍스트는 생성될 때 두 단계를 거칩니다.

![실행 컨텍스트의 생성 단계와 실행 단계](/assets/posts/js-execution-context-phases.svg)

### 생성 단계 (Creation Phase)

코드가 실제로 실행되기 전, 엔진이 준비 작업을 합니다.

- `var`로 선언된 변수를 메모리에 등록하고 `undefined`로 초기화합니다.
- `function` 선언을 메모리에 완전히 올립니다 (함수 전체가 즉시 사용 가능).
- `let`과 `const`를 메모리에 등록하지만 초기화는 하지 않습니다. **TDZ(Temporal Dead Zone, 임시 사각지대)** 상태가 됩니다.
- `this` 바인딩을 결정합니다.

### 실행 단계 (Execution Phase)

코드를 위에서 아래로 한 줄씩 실행합니다. 변수에 값을 할당하고, 표현식을 평가하고, 함수를 호출합니다.

---

## 호이스팅의 진짜 원인

호이스팅은 "코드가 위로 올라간다"는 게 아닙니다. **생성 단계에서 선언이 미리 처리된다**는 뜻입니다.

```js
console.log(a); // undefined — 생성 단계에서 이미 메모리에 있음
var a = 10;
console.log(a); // 10

greet(); // "안녕!" — 함수 전체가 생성 단계에서 올라와 있음
function greet() { console.log("안녕!"); }
```

반면 `let`과 `const`는 생성 단계에서 등록은 되지만 초기화되지 않습니다. 선언 코드 줄에 도달하기 전까지 TDZ 상태이며, 이 구간에서 접근하면 `ReferenceError`가 발생합니다.

```js
console.log(b); // ReferenceError: Cannot access 'b' before initialization
let b = 20;
```

변수 `b`는 메모리에 이미 존재합니다. 하지만 초기화되지 않은 상태로 접근을 막는 것이 TDZ입니다. `var`의 조용한 `undefined`보다 훨씬 명확한 오류 신호를 줍니다.

## 렉시컬 환경과 외부 참조

각 실행 컨텍스트는 **렉시컬 환경(Lexical Environment)** 을 가지며, 여기에는 현재 스코프의 변수들과 함께 **외부 렉시컬 환경에 대한 참조(outer)** 가 포함됩니다.

```js
const x = 1;

function outer() {
  const y = 2;

  function inner() {
    const z = 3;
    console.log(x, y, z); // 1, 2, 3
  }

  inner();
}

outer();
```

`inner`의 렉시컬 환경에 `z`가 있고, outer 참조가 `outer`의 렉시컬 환경을 가리킵니다. `outer`의 환경에는 `y`가 있고, 그 outer 참조는 전역 환경을 가리킵니다. `x`는 전역 환경에 있습니다.

이것이 스코프 체인의 실체이고, 클로저가 바깥 변수를 기억할 수 있는 이유입니다. 내부 함수의 렉시컬 환경이 외부 환경을 참조로 붙잡고 있는 한, 외부 함수가 반환된 뒤에도 그 환경은 사라지지 않습니다.

## this 바인딩이 결정되는 시점

생성 단계에서 `this`가 결정된다고 했습니다. 구체적으로는 다음과 같습니다.

- 전역 컨텍스트: 브라우저에서는 `window`, Node.js에서는 `global` (엄격 모드에서는 `undefined`)
- 함수 컨텍스트: 호출 방식에 따라 결정 — 이전 글에서 자세히 다뤘습니다.
- 화살표 함수: 자체 실행 컨텍스트를 만들지 않으므로 `this`를 갖지 않습니다. 외부 렉시컬 환경의 `this`를 그대로 사용합니다.

화살표 함수에 자체 `this`가 없다는 것도 실행 컨텍스트로 설명됩니다. 화살표 함수는 일반 함수처럼 독립적인 컨텍스트를 생성하지 않기 때문에 `this`를 스코프 체인으로 탐색합니다.

## 스택 오버플로우

재귀 함수가 종료 조건 없이 계속 호출되면 콜 스택이 꽉 찹니다. 브라우저나 Node.js는 최대 스택 깊이 제한이 있으며 초과하면 `Maximum call stack size exceeded` 오류가 발생합니다.

```js
function endless() {
  return endless(); // 종료 조건 없음
}
endless(); // RangeError: Maximum call stack size exceeded
```

이 오류를 보면 "콜 스택이 가득 찼다", 즉 실행 컨텍스트가 pop되지 않고 계속 쌓였다는 뜻으로 해석할 수 있습니다.

실행 컨텍스트는 JavaScript 런타임의 핵심 메커니즘입니다. 호이스팅, 스코프, 클로저, `this`가 모두 이 구조 위에서 동작합니다. 다음 글에서는 **이터러블과 제너레이터**를 살펴봅니다. 반복을 추상화하는 프로토콜과, 실행을 일시정지할 수 있는 제너레이터 함수가 어떻게 동작하는지 알아보겠습니다.
